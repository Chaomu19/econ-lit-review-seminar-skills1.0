# 组会 PPT 工作流

## 目标

先撰写 `PPT完整内容.md`，再生成初版 PPT，并通过人工审核与自动化批量修改完成迭代。

## 主流程

1. 确认论文池和转换后的 Markdown 文件。
2. 生成综述大纲和论文优先级表。
3. 按板块撰写 `PPT完整内容.md`。
4. 用户审核实际展示的文字、图表和表格。
5. 根据模板生成初版 PPT。
6. 用户人工修改有代表性的重复页型。
7. 用自动化代码把同类修改应用到后续类似页面。
8. 最后一轮人工打磨。
9. 为高风险论文准备兜底精读笔记。

## 推荐 PPT 结构

1. 研究背景与汇报定位。
2. 核心问题与整体逻辑。
3. 研究板块一。
4. 研究板块二。
5. 研究板块三。
6. 研究板块四。
7. 本土情境或应用边界。
8. 方法谱系、研究缺口与选题启发。

每个研究板块通常包括：

- 板块首页。
- 板块总览表。
- 精讲论文页。
- 可选的争议或过渡页。

## `PPT完整内容.md` 标准

每页应包括：

- 页码。
- 页面标题。
- 核心内容。
- 如使用图表，标明图表来源。
- 本页关键判断。
- 可选口播稿。

如果论文数量较多，应按板块逐步生成，不要一次性生成整份 PPT。

## 用户审核重点

用户应重点审核：

- 每页是否值得展示。
- 所选图表是否必要。
- 文字是否过密。
- 页面是否支撑主论证。
- 是否有论断需要复核。

这个审核过程本身也是高效阅读过程：只优先阅读真正会展示的内容。

## PPT 自动化

如果用程序生成 `.pptx`：

- 使用可复现的源文件，例如基于 PptxGenJS 的 JavaScript。
- 路径必须可配置。
- 不提交 `node_modules`。
- 默认不提交生成的 `.pptx`，除非用户明确希望将其作为发布产物。
- 只有在模板可以共享时，才把用户模板放入仓库。

对于重复页型：

- 让用户先人工修改一页代表性页面。
- 检查修改后的版式或源文件。
- 用代码将该模式应用到同类页面。
- 再检查代表性页面。
